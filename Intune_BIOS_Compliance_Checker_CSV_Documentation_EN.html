BIOS Compliance Checker CSV Mode v1.11

A login-free, fully CSV-based variant of the Intune BIOS Compliance Checker. Instead of fetching data live via the Microsoft Graph API, two CSV exports from the Intune Admin Center are imported. No Azure AD, no app registration, no token — the browser processes everything locally.

What is this tool?

The tool checks Windows devices from Intune for BIOS compliance with respect to the Windows Secure Boot 2023 certificate update (expiring June 2026). It compares the installed BIOS version against the minimum versions published by Dell, HP, and Lenovo.

Unlike the Graph-based variant, this tool requires no sign-in and no Azure AD app registration. Device data is loaded instead from a CSV export from the Intune Admin Center.

Export the device CSV

The CSV export is done in the Intune Admin Center:

1

Open Intune Admin Center

Navigate to intune.microsoft.com → Devices → All devices.

2

Start export

Click Export (top right of the device list). Intune creates a full export of all devices including hardware inventory data.

3

Download CSV

The export is provided as a ZIP file. Load the contained CSV file directly into the tool.

⚠️
Hardware inventory must be enabled

The SystemManagementBIOSVersion column is only populated if hardware inventory is enabled for devices in Intune and devices have already reported their data. Devices without a BIOS value are shown as Check manually.

Export the Secure Boot CSV

Optionally, a second CSV export from Intune Reporting can be imported. It contains Secure Boot status, certificate status, and — from the updated report as of May 2026 — Confidence Level, Secure Boot Trust configuration and the date last reported.

Intune Admin Center → Reports → Windows Autopatch → Windows quality updates → Reports → Secure Boot status → Export

The tool automatically links the data via device name (case-insensitive). Columns show — until a CSV is imported.

⚠️
Columns in the report export

The current Intune export of the Secure Boot Status report only contains the base columns (Device name, Secure Boot enabled, Firmware version). The new columns (Certificate status, Confidence level, Secure Boot trust configuration, Date last reported) are visible in the report UI but are not yet included in the export. The tool is ready and will display the data once Microsoft updates the export.

CSV import

The tool provides two drop zones side by side:

  • Device CSV (required) — via drag & drop or click
  • Secure Boot Status CSV (optional) — via drag & drop or click

Both CSVs can be imported in any order. If the Secure Boot CSV is loaded after the analysis is started, the table updates automatically.

ℹ️
Automatic filtering during import

The following devices are automatically excluded during parsing and do not appear in the results table:

  • Non-Windows devices — column OS does not contain Windows
  • MDE-only devices — column Managed by = MDE
  • Personal devices — column Ownership = Personal
  • Windows Home SKU — column SkuFamily contains home
  • VMware VMs — manufacturer or model contains vmware (case-insensitive)
  • Microsoft Hyper-V/Azure VMs — manufacturer = Microsoft Corporation AND model contains Virtual Machine (Surface devices are retained)

Run analysis

After importing the device CSV, the Run analysis button becomes active. Clicking it:

  1. Populates the filter dropdowns with available vendors and models
  2. Runs the BIOS compliance check for all devices
  3. Shows the results table and summary tiles
  4. Automatically scrolls to the results

Reset clears all imported data and returns the tool to its initial state.

CSV parsing logic

Device CSV

The tool uses a complete RFC 4180-compatible CSV parser that handles quoted fields, embedded commas and line breaks. The following are automatically filtered during parsing:

  • Non-Windows devices — column OS must contain Windows
  • MDE-only devices — column Managed by = MDE
  • Personal devices — column Ownership = Personal
  • Windows Home SKU — column SkuFamily contains home
  • VMware VMs — manufacturer or model contains vmware
  • Microsoft Hyper-V/Azure VMs — manufacturer = Microsoft Corporation AND model contains Virtual Machine

The SystemManagementBIOSVersion column is detected by searching for the keyword bios in the header — regardless of capitalisation.

Secure Boot CSV

The Secure Boot CSV is also parsed RFC 4180-compatibly. The link to device data is made via device name (column Device name) as the join key, case-insensitive. Required fields:

  • Device name — join key (required)
  • Secure Boot enabled — Yes / No / Unknown (required)

Optional columns — automatically recognised if present, missing field shows —:

  • Certificate status — Up to date / Not up to date / Not applicable
  • Confidence level — free text, normalised internally to High / Under Observation / No Data / Paused / Not Supported
  • Secure Boot trust configuration — free text
  • Date last reported — date as text

BIOS compliance check

Version comparison

The installed BIOS version is compared segment by segment numerically against the minimum version — not alphabetically:

compareVersions("01.10.00", "01.09.00")  →  1  (current ≥ minimum ✓)
compareVersions("1.8.0",    "1.10.0")    → -1  (outdated ✗)
compareVersions("1.10.0",   "1.10.0")   →  0  (exactly equal ✓)

Model matching (normalisation + fuzzy)

Intune often returns model names in verbose form. Matching runs in three stages:

StageMethodExample
1 Normalisation Remove noise words (Notebook PC, Inch, Laptop PC, …) HP EliteBook 860 16 inch G10 Notebook PC → hp elitebook 860 16 g10
2 Substring Normalised DB key is contained in Intune name (or vice versa) hp elitebook 840 g8 ⊂ hp elitebook 840 g8 special config
3 Fuzzy (Jaccard) Word-set similarity ≥ 60% hp elitebook 840-g8 ↔ hp elitebook 840 g8

A tooltip on the minimum version cell shows which DB entry was used for the match (e.g. t14 gen 1 (MT:20S1)).

BIOS version extraction

The tool recognises three BIOS version formats:

// HP: family prefix + "ver." + version
"V70 ver. 01.10.00"       →  "01.10.00"
"T76 Ver. 01.22.00"       →  "01.22.00"

// Lenovo ThinkPad/ThinkBook (commercial): version in parentheses
"N2XET42W (1.32 )"        →  "1.32"
"R1EET56W(1.56 )"         →  "1.56"
"F8CN59WW(V2.22)"         →  "2.22"
"MNCN37WW (1.37)"         →  "1.37"   (ThinkBook)

// Lenovo IdeaPad/Legion/Yoga (consumer): no version format
"GLCN46WW"                →  not extractable → Check manually
"Q8CN17WW"                →  not extractable → Check manually

// Dell / generic: already numeric
"01.10.00"                →  "01.10.00"  (unchanged)
"1.32.1"                  →  "1.32.1"    (unchanged)

The original raw value is exported as a separate column in the CSV and shown as a tooltip on the BIOS cell — provided the device is verifiable.

Lenovo machine type detection

Intune reports machine type codes for Lenovo devices (e.g. 20S1SFT70Y or 83LKS0B100), not the product name. Normal fuzzy search would find nothing in the DB. The tool therefore uses a separate lookup table (LENOVO_MT) with 463 entries from HT518129:

StepActionExample
1. Fuzzy matchNormal DB lookup (as for Dell/HP)Fails for machine type codes
2. BIOS family disambiguationComposite key MT_BIOSPREFIX in LENOVO_MT_BIOS20VX_N3WET → min 1.13 instead of 1.61
3. Machine type lookupFirst 4 characters of model code in LENOVO_MT20S1 → t14 gen 1 → min 1.31
4. Desktop detectionPrefix 10–13xx, 30xx, 90/91xx, F0xx30CE → ThinkStation → Check manually ⓘ
5. Consumer detectionPrefix 82xx or 83xx without MT entry83LK → IdeaPad → Check manually ↗
ℹ️
BIOS family disambiguation (P14s/P15s Gen 2)

Some Lenovo models ship with two different BIOS families with different minimum versions. The tool identifies the BIOS family from the first 5 characters of the raw value and selects the correct minimum: 20VX with N34ET BIOS → min 1.61, with N3WET BIOS → min 1.13.
