Intune BIOS Compliance Checker v2.0

Ein browserbasiertes Tool zur Übersicht aller Windows-Geräte in Microsoft Intune mit Abgleich der aktuellen BIOS-Version gegen die von Dell und HP veröffentlichten Mindestversionen für das Windows Secure Boot 2023 Zertifikats-Update (Ablauf Juni 2026). Optional erweiterbar um Secure Boot Status und Zertifikat-Status per CSV-Import aus dem Intune-Portal.

Was ist das Tool?

Die Microsoft Secure Boot Zertifikate aus dem Jahr 2011 laufen ab Juni 2026 aus. Damit Windows-Geräte weiterhin Secure-Boot-bezogene Sicherheitsupdates erhalten können, müssen OEM-Hersteller BIOS-Updates mit den neuen 2023-Zertifikaten bereitstellen — und diese müssen auf den Geräten installiert sein.

Dieses Tool liest alle Windows-Geräte aus Intune via Microsoft Graph API aus, holt die BIOS-Version per Einzelabruf ab und vergleicht sie automatisch mit den veröffentlichten Mindestversionen der Hersteller. Das Ergebnis ist eine filterbare, exportierbare Compliance-Übersicht.

Voraussetzungen

  • Moderner Browser (Chrome, Edge, Firefox, Safari)
  • Zugriff auf einen Microsoft Entra ID (Azure AD) Tenant
  • Berechtigung zum Erstellen einer App-Registrierung (oder eine bereits vorhandene)
  • Die App-Registrierung benötigt die delegierte Berechtigung DeviceManagementManagedDevices.Read.All
  • Ein Intune-Administrator-Account für die erste Anmeldung (Admin Consent)

Berechtigungen konfigurieren

In der App-Registrierung unter API-Berechtigungen:

  1. Klick auf Berechtigung hinzufügen → Microsoft Graph
  2. Typ: Delegierte Berechtigungen
  3. Suche nach DeviceManagementManagedDevices.Read.All und aktiviere die Checkbox

Wichtig: Delegierte Berechtigung, nicht Application

Die Berechtigung muss als delegierte Berechtigung eingetragen sein — nicht als Anwendungsberechtigung. Ohne Admin Consent wird beim ersten Login ein Zustimmungsdialog angezeigt.

Erste Anmeldung

Nach dem Öffnen des Tools im Browser:

  1. Tenant ID und Client ID in die Felder eingeben
  2. Klick auf Mit Microsoft anmelden — es öffnet sich ein Popup-Fenster
  3. Mit einem Intune-Leseberechtigten Account anmelden
  4. Beim ersten Mal ggf. den Zustimmungsdialog bestätigen
  5. Das Tool beginnt automatisch mit dem Datenabruf

Tenant ID und Client ID werden in der sessionStorage gespeichert und stehen beim nächsten Öffnen im selben Tab wieder vor. Gibt es noch keinen gecachten Account, wird direkt das Login-Popup geöffnet — kein stiller Token-Versuch, der im file://-Kontext scheitern würde.

Datenabruf

Schritt 1 — Geräteliste

Das Tool ruft alle verwalteten Windows-Geräte ab — MDE-only Geräte (managementAgent = msSense) werden serverseitig ausgeschlossen:

GET /beta/deviceManagement/managedDevices
    ?$select=deviceName,manufacturer,model,operatingSystem,osVersion,id
    &$filter=operatingSystem eq 'Windows'
      and (managementAgent eq 'mdm' or managementAgent eq 'easMdm'
           or managementAgent eq 'intuneClient' ...)
    &$top=999

Bei mehr als 999 Geräten werden automatisch alle Seiten via @odata.nextLink abgerufen.

Schritt 2 — BIOS-Daten pro Gerät

Da hardwareInformation im List-Endpunkt immer leer zurückgegeben wird, erfolgt ein Einzelabruf pro Gerät in Batches von 10 parallelen Requests:

GET /beta/deviceManagement/managedDevices/{id}
    ?$select=hardwareInformation
⏱️
Ladezeit

Bei 500 Geräten sind das ~50 Runden à 10 parallele Anfragen. Die Ladezeit beträgt je nach Tenant-Grösse und Netzwerk 1–3 Minuten. Der Fortschritt wird laufend angezeigt.

BIOS-Compliance-Prüfung

Versionsvergleich

Die aktuelle BIOS-Version wird segmentweise numerisch mit der Mindestversion verglichen — nicht alphabetisch:

compareVersions("01.10.00", "01.09.00")  →  1  (aktuell >= minimum ✓)
compareVersions("1.8.0",    "1.10.0")    → -1  (veraltet ✗)
compareVersions("1.10.0",   "1.10.0")   →  0  (exakt gleich ✓)

Modellabgleich (Normalisierung + Fuzzy)

Intune liefert Modellnamen oft in ausführlicher Form. Der Abgleich läuft in drei Stufen:

StufeMethodeBeispiel
1 Normalisierung Störwörter entfernen (Notebook PC, Inch, Laptop PC, …) HP EliteBook 860 16 Inch G10 Notebook PC → hp elitebook 860 16 g10
2 Substring Normalisierter DB-Schlüssel ist in Intune-Namen enthalten (oder umgekehrt) hp elitebook 840 g8 ⊂ hp elitebook 840 g8 sonderconfig
3 Fuzzy (Jaccard) Wortmengen-Ähnlichkeit ≥ 60 % hp elitebook 840-g8 ↔ hp elitebook 840 g8

Ein Tooltip auf der Mindestversions-Zelle zeigt an, welcher DB-Eintrag für den Match verwendet wurde.

BIOS-Versionsextraktion

HP gibt BIOS-Versionen in Intune mit der BIOS Family als Präfix zurück. Das Tool extrahiert die numerische Version automatisch:

"V70 ver. 01.10.00"  →  "01.10.00"
"T76 Ver. 01.22.00"  →  "01.22.00"
"01.10.00"           →  "01.10.00"   (unverändert)
"1.32.1"             →  "1.32.1"     (unverändert)

Der ursprüngliche Rohwert aus Intune ist als Tooltip auf der BIOS-Zelle sichtbar und wird im CSV-Export als separate Spalte ausgegeben.

Secure Boot CSV-Import

Secure Boot Status und Zertifikat-Status sind über die Graph API nicht direkt abrufbar — sie stehen nur im Intune-Portal-Report zur Verfügung. Das Tool unterstützt daher einen optionalen CSV-Import.

CSV exportieren

Im Intune Admin Center: Reports → Windows Autopatch → Windows quality updates → Reports → Secure Boot status → Exportieren

CSV importieren

Nach dem Login erscheint eine CSV-Import-Karte. Die CSV-Datei per Drag & Drop ablegen oder über den Klick-Dialog auswählen. Das Tool verknüpft die Daten automatisch über den Gerätenamen (case-insensitive).

ℹ️

Das Tool ist ohne CSV vollständig nutzbar. Die Secure Boot- und Zertifikat-Spalten zeigen —, solange kein CSV importiert wurde.

Gerätfilterung

Das Tool filtert serverseitig auf Windows-Geräte und schliesst MDE-only Geräte aus. Da der OData ne-Operator für Enum-Felder nicht unterstützt wird, verwendet das Tool eine Allowlist mit eq/or:

$filter=operatingSystem eq 'Windows'
  and (managementAgent eq 'mdm'
       or managementAgent eq 'easMdm'
       or managementAgent eq 'intuneClient'
       or managementAgent eq 'easIntuneClient'
       or managementAgent eq 'configurationManagerClientMdm'
       or managementAgent eq 'configurationManagerClientMdmEas'
       or managementAgent eq 'microsoft365ManagedMdm'
       or managementAgent eq 'intuneAosp')

Warum Beta-Endpunkt?

hardwareInformation.systemManagementBIOSVersion ist nur im /beta-Endpunkt verfügbar. Im stabilen /v1.0-Endpunkt existiert hardwareInformation nicht als selektierbares Feld.

Warum Einzelabruf statt Liste?

hardwareInformation wird beim Listen-Endpunkt grundsätzlich leer zurückgegeben. Nur der Einzelabruf via /managedDevices/{id} liefert das befüllte Objekt.
